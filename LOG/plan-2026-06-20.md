This session processed the two captures filed during recent on-device verification and rolled a now-runnable deferred test. Both captures were small, dependency-free refinements to the Schedule/navigation surface, each promoted as a single batch with the spec-edit folded in beside the code.

**Tomorrow date label.** Drop the DD/MM label from Tomorrow, leaving dates only on Soon and Later. The call rested on a detail stronger than the capture stated: a Tomorrow task always carries exactly tomorrow's date — the slot is derived from the date, so nothing else can land there — making the label fully redundant with the page title. And Tomorrow has no past-date case the way Today does (past-dated tasks fall onto Today), so dropping the label loses no stale-date signal. Today's rule is untouched.

**Swipe-to-open menu.** Disable the navigation drawer's left-edge swipe-to-open so the ☰ button is the only opener. The decision turned on Taskflow's signature gesture — horizontal swipe drives the whole spine — which the drawer's edge-swipe collides with, giving the same gesture region-dependent meaning. The Android edge-swipe-to-open convention was the weighed counterweight and was set aside: it carries less weight in an app that repurposes horizontal swipe as core navigation, and the ☰ remains a standard, discoverable opener, so no affordance is truly lost. A web search on Material 3 drawer-gesture guidance was offered and declined — decided on reasoning. Implementation verified in AppRoot.kt: `gesturesEnabled = false` disables only the drawer's gesture open/close; the programmatic ☰ (`drawerState.open()`) and the spine's `HorizontalPager` are unaffected. Known side effect recorded in the batch: swipe-to-close also goes, but scrim-tap and item-tap still close the drawer.
