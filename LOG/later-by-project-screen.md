# [HASH] — Later slot rebuilt as Project-grouped expand/collapse cards (new `ui/schedule/LaterPage.kt`; one per Project, Unassigned pinned bottom, collapsed by default; within-card drag-reorder split out to a capture); deleted the per-Project view (`ui/project/`) and Projects overview (`ui/projects/`) plus their nav destinations, spine page, overlay host, drawer per-Project list, and the now-unused `EditTarget.NewInProject`; editor Project picker confirmed as the move-between-Projects path [later-by-project-screen]

The visible half of the Later-by-Project redesign. Later stops listing far-future tasks by date and becomes a vertical list of expand/collapse cards — one per Project (every Project shown, even empty), the system Unassigned card pinned at the bottom — each holding that Project's far-future dated tasks (DD/MM) and its undated tasks. This removes the "tapping a Project whisks you off the spine" feel: Projects no longer have a surface of their own, so reaching one means opening Later and expanding its card. Getting there meant deleting the per-Project view (`ui/project/`) and the Projects overview page (`ui/projects/`), plus the Projects spine page, the Project overlay and its host in AppRoot, the drawer's per-Project list and teaching blurb, and the Project/overview destinations. The grouping is driven from the schedule layer: ScheduleViewModel now also observes the ordered Projects and groups Later tasks into one card per Project in Strategy-doc order, Unassigned last. Later membership is far-future dated OR undated-not-parked-on-Soon — which now correctly includes never-slotted undated tasks that the old flat Later bucket silently dropped.

Within-card drag-to-reorder, named in the batch, was split out to a capture mid-build with Alex's approval. It is a from-scratch reorderable list — no drag pattern exists in the app yet and no library — and it shares its whole mechanism with [task-reorder-within-list] (slot reorder, still unbuilt), so building it in both batches separately would invent the same primitive twice. The cards ship without reorder (a coherent partial); the reorder is designed once, with [task-reorder-within-list], in /plan.

A consequence surfaced on device and is worth recording plainly: deleting the Projects overview removed the only Project-creation UI (its "+ New Project" button). Its replacement — a "New Project" entry in the editor's Project picker — is [project-lifecycle-later], which couldn't start until this batch landed. So this build leaves a window with no way to create a Project at all. Planning anticipated this (the [project-lifecycle-later] note says the old button "goes away with that page"), and AppViewModel.createProject was preserved — now caller-less — for that batch to wire up. Because no user Project can be created, the device checks needing one (move a task between Projects; several user-Project cards in Strategy order; far-future-dated under a user Project) couldn't run and are deferred. [project-lifecycle-later] is now unblocked and should be prioritised so the no-create window is short.

On device, Alex asked that cards open showing the first few tasks rather than collapsed. That reverses the recorded collapsed-by-default decision, so it was not changed here — it is captured for a /plan spec-edit plus a follow-up build. The number was weighed: 7 was floated, ~3 chosen, because ~7 per card across several Projects rebuilds the wall of tasks that collapsed-by-default existed to prevent, while ~3 is a glanceable peek.

Verified: `compileDebugKotlin` and `assembleDebug` both green, no dangling references to the deleted screens; installed via adb over wireless (with permission) and device-tested with Alex — card structure, Unassigned pinned, empty card, overview gone, slim side menu (Today · Tomorrow · Soon · Later · Strategy), and undated-add-on-Later → Unassigned card all passed.
