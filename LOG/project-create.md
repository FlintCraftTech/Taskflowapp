# d0d4f89 — ProjectsOverviewScreen: added manual name-only Project creation — a "+ New Project" affordance + name dialog wired through ScheduleScreen/AppRoot to a new AppViewModel.createProject that inserts at next sort-order and creates the Project's empty Strategy entry — [project-create]

Until now nothing in the build could create a Project — the overview and side menu only listed existing ones, no UI called insert, and the device DB stayed empty, so the Project view, Strategy doc, and Project-assignment all had nothing to work with. This batch adds the manual path SPEC §Projects overview already describes: an add affordance on the Projects overview that takes a name and creates the Project at the end of the sort order (max + 1), creating its empty Strategy entry alongside so the Strategy-doc paragraph exists to be written later. Name-only by design — a Project's description is its Strategy paragraph, written later, so asking for one here would duplicate that editor.

Two things were learned in the build. First, the add affordance lives inside the Projects-overview content, but that content renders through ScheduleScreen (the spine pager), which the batch hadn't listed — so ScheduleScreen was added to scope (user-approved) to thread one createProject callback through. The alternative, a floating + button placed by AppRoot, was weighed and set aside: it avoided the extra file but moved the affordance out of the overview screen and changed it to a FAB, diverging from the batch's design. Second, the data layer needed no changes — getMaxSortOrder(), insert(), and upsert() already existed, so the batch's "add a max-sort-order query if absent" contingency didn't fire. The name dialog was inlined as a private composable rather than its own file. The "+ New Project" affordance also shows in the empty Projects state, since that's exactly when the first Project gets made.

Creation was verified on-device (✓): a typed name creates a Project that appears on the overview, in the side menu last position, and as a selectable Project in the task editor. The two folded-in Project-view checks ([0004] rendering, [0005] add/refile) were deferred: while testing, Alex raised two design objections about Projects and stopped verification. Both were routed to Captures for /plan, not acted on. One asks whether Projects should appear in the side menu at all; the other asks how a Project should be reached, since tapping one opens an off-spine surface inconsistent with every other spine destination. The second was flagged against the prior decision (2026-06-20, [later-by-project]) that the per-Project view survives the merge as the home for undated tasks — it does not reopen that; it questions navigation to the surface, not the surface's existence.
