# [HASH] — Schedule view (batch 0002) — built the read-only Today/Tomorrow/Soon/Later screens as a HorizontalPager (spine's left segment, Today default) with date-derived slot placement (new domain/SlotDeriver + 11 passing unit tests, 4 AM day-start constant pending 0012), per-slot empty states, and a new TaskDao/TaskRepository active-tasks Flow; header redesigned mid-build (user-directed) from an all-tabs row to a single centred current-slot name that slides in the swipe direction, flanked by tappable ←/→ chevrons that hide at dead ends, corners left clear for the future menu key + pick-up delete target; verified on a Pixel 6. → 0002-schedule-view-date-derived-placement.md

**What shipped.** The read-only Schedule view. The four slots — Today, Tomorrow, Soon, Later — are pages on a HorizontalPager, the left segment of the navigation spine, with Today the default page. Placement is date-derived: a new pure `SlotDeriver` buckets a task's date relative to "now" (today and past → Today, +1 → Tomorrow, 2–7 → Soon, 8+ → Later), shifted by a day-begins-at boundary. The boundary is a 4 AM constant for now; 0012 wires the user setting. Undated tasks parked on Soon/Later show by their slot column. Rows are read-only and show a DD/MM date on Tomorrow/Soon/Later, and on Today only when the date has slipped into the past. Each empty slot shows a plain placeholder; final copy is the parked empty-state polish capture. A new `TaskDao.observeActiveTopLevel()` Flow (+ repository passthrough) feeds the bucketing. `SlotDeriver` has 11 JVM unit tests, all passing.

**Decisions taken and why.** Header redesigned mid-build at the user's direction: the first cut used a Material tab row showing all four names; the user wanted only the current slot's name, centred, with an arrow either side to hint the screen slides. Final header is a single name in a fixed frame sized to the longest label ("Tomorrow") so it doesn't reflow, sliding in the swipe direction via AnimatedContent, flanked by tappable ←/→ chevrons that hide at a dead end (no left arrow on Today, no right on Later). The corners are left clear on purpose — the side-menu key (top-left) and the pick-up delete target (top-right) land there in later batches. This removed the experimental-API opt-in the tab row needed. Rollover is treated as derivation, not mutation: because placement is computed from the date, Tomorrow's tasks fall into Today on their own once the boundary passes; the append-to-bottom reorder on rollover is left to 0012, and 0002 renders by stored sort order with a date+id tiebreak. Read-only was kept strict (no add/edit/drag/complete, no completed tray — those are later batches). Dates are stored as epoch millis (matching 0001's tests); java.time is used (minSdk 26).

**Pivots and surprises.** The "no device on this machine" assumption was wrong — the user connected a Pixel 6 over wireless adb mid-session, so the screen was verified on real hardware: render, Today as default, swipe and arrow-tap navigation, dead-end arrow hiding, and empty states all confirmed. Data-dependent rendering still can't be exercised (no way to add tasks until 0005). Two UI bugs were caught on-device and fixed: the ❮/❯ ornament glyphs weren't in the device font and rendered blank (switched to ←/→), and a `fillMaxWidth` on the title pushed the right arrow off-screen (removed, restoring the fixed frame). The Gradle "unable to delete build" file-lock hit when the user pressed Run in Android Studio (its file-watching holds the lock); cleared with `--stop` + deleting `app\build`, then built and installed from the CLI with the `--no-watch-fs --no-daemon` flags.
