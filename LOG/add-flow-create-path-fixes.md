# 55d7165 — Add-flow create-path fixes — fixed the stale New-task title (EditTaskScreen.kt: per-slot ViewModel key in the Activity store replaced with a dialogue-scoped ViewModelStore cleared on dispose, so each FAB open opens a fresh blank form), and closed the Today-slot "stores tomorrow's date" bug as not-a-bug — verified correct by live device test plus static proof (a Today-add stores noon today, which can't cross a day), no code change; filed a capture proposing Tomorrow drop its date label

This batch carried two device-verify findings from the 0005 add flow. One was real and fixed; the other could not be reproduced and is now verified correct.

**Bug 2 — stale New-task title — was real and is fixed.** The edit dialogue keyed its view-model by slot (`edit-new-slot-TODAY`), and that view-model lived in the Activity's ViewModelStore, so it survived the dialogue closing. A second add on the same slot reused the same instance, with the previous add's title still in its form. The fix gives each dialogue open its own store: a remembered `ViewModelStoreOwner` passed to `viewModel(...)` and cleared in `DisposableEffect.onDispose`. EditTaskScreen leaves and re-enters the composition on each open, so the store and its view-model are rebuilt every time — a fresh blank form per add. The batch suggested a per-open key instead; that was rejected because a monotonic key would pile up one view-model per add in the Activity store (a slow leak), while the scoped store gives the same fresh-form outcome with no leak. The now-dead `viewModelKey()` helper was removed. `compileDebugKotlin` succeeded.

**Bug 1 — Today-slot add storing tomorrow's date — could not be reproduced; closed as not-a-bug, no code change.** A full static trace of the FAB→slot→date chain confirmed the original capture's own conclusion: the Today path stores noon of today, and noon sits eight hours clear of the 4 AM day boundary, so a Today-add can never compute or store tomorrow (the day-diff is always 0 → TODAY). The dialogue's date label, the schedule bucketing, and the page→slot map are all statically correct too. A live test on the Pixel confirmed it end to end: on the Today page, tapping + opened a dialogue showing today's date; saving landed the task on Today with no label. The earlier "stored tomorrow's date, confirmed against SQLite" report could not be reconstructed; the most likely explanation is that the raw epoch value was read in the wrong timezone. Closed on strong evidence — a live test plus a proof that the bug is impossible in the current code — with the caveat, stated to Alex and accepted by her, that the original observation was never fully explained.

The Today-add test also surfaced a design idea, filed as a capture: drop the date label from Tomorrow as well (Today and Tomorrow don't need a date — the page name says the day; dates earn their place on Soon and Later). That is a change to the spec as written, not a bug, so it goes through /plan.
